Video Cloud HLS 암호화 란 무엇입니까?

HTTP Live Streaming Encryption (HLSe)을 사용하는 경우 콘텐츠는 AES 128 표준으로 암호화되며 매니페스트에는 HLS 콘텐츠의 암호 해독을 허용하는 키에 대한 링크가 포함됩니다. 이것만으로는 완전한 보안을 제공하지 않습니다. 키를 얻은 후에는 콘텐츠를 쉽게 해독하고 재배포 할 수 있기 때문입니다. HTTP 또는 다른 토큰 인증 모델을 통해 제공하는 것과 같이 키 보호를 허용하는 메커니즘은 거의 없습니다. HLSe는 콘텐츠를 가져 오려는 대부분의 표준 사용자로부터 콘텐츠를 보호하지만 DRM 수준 콘텐츠 보호로 간주되지 않습니다. Video Cloud는 Brightcove 플레이어에서 HLS 암호화를 지원합니다. 브라이트코브 플레이어 지원에 대한 자세한 내용은브라이트코브 플레이어 시스템 요구 사항을 참조하십시오 . 또한 아래의제한 사항섹션을 확인하십시오.

Video Cloud Enterprise는 HLS 암호화를 사용하여 콘텐츠를 어떻게 보호합니까?

암호화와 독립적 인 Apple HTTP Live Streaming (HLS)은 다른 비트 전송률을 사용하는 비디오 제공 프로토콜입니다. Video Cloud는 네트워크 대역폭 변경 및 서비스 변동에 따라 변환간에 지능적으로 전환하는 다중 변환 생성을 지원합니다. HLS는 기본적으로 비디오를 일련의 작은 파일 다운로드로 나누고 각각 HTTP를 통해 한 번에 비디오의 짧은 청크 또는 세그먼트를로드합니다.

참고: Apple은 긴 형식의 비디오, 즉 5MB 이상 또는 10 분 이상의 비디오에 HLS를 요구합니다. 암호화에 관계없이 iOS 기기에서 약 10분 이상 동영상을 재생하려면 HLS 변환을 만들어야 합니다.

Video Cloud는 Apple HLS 용 비디오 변환 암호화를 지원하므로 게시자가 HLS를 통해 장치에 전달되는 긴 형식의 비디오 콘텐츠를 보호 할 수 있습니다. Video Cloud HLS 암호화는 표준 HLS 솔루션에 AES를 추가하여 콘텐츠를 보호합니다. Apple HLS에 대한 암호화를 구현할 때 Video Cloud는 비디오의 각 작은 파일 세그먼트를 암호화하고 변환 선택을 처리하는 파일을 안전하게 제공합니다.

지원되는 기능

전자 데이터 암호화를 위해 AES 사양을 활용하는 것 외에도 Video Cloud HLS 암호화는 다음과 같은 방식으로 콘텐츠를 추가로 보호합니다.

  • 각 세그먼트 파일은 암호화됩니다.
  • Video Cloud에서 제공하는 HLS 매니페스트 (.m3u8 파일)에는 각 세그먼트의 키에 대한 링크가 포함되어 있습니다.
  • 기본적으로 키는 10분마다 회전되며 사용자 정의 인제스트encryption_key_rotation_period프로파일에서 를 지정할 수 있습니다.
  • 동적전달이 활성화된 계정에 대한 HLS 변환에 암호화를 추가하려면 브라이트코브 지원에 요청을 제출하면 계정에 대한 HLSE를 활성화할 수 있습니다. (HLSe에 대해 이미 활성화 된 계정의 경우 계정에 Dynamic Delivery가 활성화 된 경우 암호화가 모든 HLS 변환에 계속 적용됩니다.)
  • 레거시 수집 시스템 (비 동적 배달) 을 사용하여 계정의 HLS 변환에 암호화를 추가하려면 각 필드에encryption_method및encryption_key_rotation_period필드를 추가합니다.수집 프로필에 정의된 HLS 변환은 다음과 같습니다. 
    {
  "media_type": "video",
  "reference_id": "ts0",
  "format": "ts",
  "type": "segmented",
  "audio_codec": "aac",
  "audio_bitrate": 64,
  "video_codec": "h264",
  "encryption_method": "aes-128",
  "encryption_key_rotation_period": 10,
  "video_bitrate": 450,
  "decoder_bitrate_cap": 771,
  "decoder_buffer_size": 1028,
  "keyframe_rate": 0.5,
  "max_frame_rate": 30,
  "width": 480,
  "height": 270,
  "h264_profile": "baseline"
참고: 비디오 암호화는 그 자체로 강력한 형태의 콘텐츠 보호가 아닙니다. 조직에 컨텐츠 보안이 중요한 경우 DRM 보호 기능을 사용해야합니다. HLS 암호화 및 DRM 보호 번들링에 대한 자세한 내용은 계정 관리자에게 문의하여 콘텐츠가 재생되는 곳마다 안전하게 보호할 수 있습니다.

HLS 암호화 구현 후 발생하는 작업

Video Cloud HLS 암호화는 각 변환 및 각 변환의 각 세그먼트가 여러 방식으로 보호되는 안전한 다중 비트 전송률 인코딩을 제공합니다. HLS 암호화 된 비디오는 비디오의 첫 번째 변환이 업로드되고 암호화 될 때 데스크톱 및 모바일 장치에서 재생할 수 있습니다. 일단 구현되면 이후에 업로드되는 모든 비디오는 HLS 암호화를 사용하여 보호됩니다. Video Cloud HLS 암호화는 장치에서 비디오 재생에 감지 할 수있는 변경 사항을 추가하지 않습니다. Video Cloud HLS 암호화는 비디오 파일의 HLS 변환에만 영향을 미치며 MP4 변환에는 영향을주지 않습니다.

한계

  • HLS 암호화는 계정의 모든 비 DRM HLS 변환에 적용됩니다. 암호화없이 제공하려는 프로모션 또는 기타 비디오가있는 경우 HLS 암호화를 활성화하지 않고 다른 Video Cloud 계정에 업로드 할 수 있습니다.
  • 레거시 수집 시스템을 사용하여 수집 된 비디오의 경우 Video Cloud는 HLS 암호화를 구현하기 전에 Video Cloud에 업로드 된 HLS 변환 암호화를 지원하지 않습니다. 이전에 업로드 된 HLS 콘텐츠는 암호화되지 않은 상태로 유지됩니다. 당신은해야합니다비디오 재변환 HLS 암호화 전에 업로드하여 보호합니다. 이것은 ~ 아니다 Dynamic Delivery 시스템을 사용하여 수집된 비디오에 적용됩니다.
  • Video Cloud는 미디어 모듈에서 HLS 암호화 된 비디오를 식별하는 표시기를 표시하지 않습니다.
  • 사용자가 Apple 장치에서 HLS 암호화 된 비디오를 재생 한 다음 TTL이 만료 된 후 재생을 시도하면 재생이 시작되지 않고 사용자에게 경고 메시지를 제공하지 않습니다.
  • Brightcove Player를 사용할 때 IE8 또는 IE9에서는 HLS 재생이 지원되지 않습니다. 이러한 장치는 MP4로 대체됩니다.
  • HLSE는안드로이드 SDK에서지원되며 이전 버전에서는 VisualON 구성 요소를 사용하여 지원됩니다. HLSe는 HTML5 플레이어와 함께 4.x에서도 작동해야합니다. Android 기기에서 HLSE 지원에 대한 자세한 내용은 Android 지원 미디어 형식을 참조하십시오 .
  • HTML5 플레이어의 경우 HLSe 지원은 기본 OS / 기기에 따라 완전히 결정됩니다.
  • Video Cloud HLS 암호화를 종료하기로 선택하면 계정의 모든 비디오에서 암호화가 제거됩니다. 다시 트랜스코딩할 필요가 없습니다.
  • 현재 HLSe 콘텐츠는 Android, iOS 또는 tvOS 용 Brightcove Native SDK를 사용한 오프라인 재생에서 지원되지 않습니다.
